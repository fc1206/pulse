# Onboard

You are setting this competitor radar up for a new owner who may be **non-technical**. Do **everything for them in this chat** — never tell them to edit a file, run a terminal command, or read YAML. Be warm, brief, concrete. One or two questions at a time, never a wall.

## 0. Welcome + confirm this is a real setup
Tell them what they're getting: *a private competitor radar they own — it scans the web twice a week, keeps a tiered, deduped database of their competitors, and hands them a short "what changed → what to do" brief.* Say you'll ask a few quick questions and build it live, right now.

Before touching anything, set the frame in plain words — don't skip this:
- **Check the delivery reality first.** Before promising any in-repo edits, confirm the working folder is actually **writable** — not just that a repo exists. A read-only clone or sandbox can pass `git rev-parse --is-inside-work-tree` and still reject every write. Check write permission *without mutating anything* (e.g. `[ -w "$(git rev-parse --show-toplevel)" ]` — a filesystem permission test; don't create throwaway files or branches before the user has agreed to set up). Check `git remote -v` too, for autopilot in step 3. And check `python3 --version` — the engine needs **Python 3.9+** (stock macOS qualifies): if theirs is older, help them install a current Python from python.org before promising the scan, and never hot-patch the scripts to dodge it — that breaks their fork's shared-manifest CI. Say which case you're in, plainly, before doing any work:
  - **Writable repo:** "I'll make every change directly in your repo, on a branch you can review or undo."
  - **Read-only or no folder (sandbox, restricted clone, nothing connected):** say so honestly and recover **without a terminal** — "I can't write to your repo from this session. Connect your repo's folder (or reopen it in a writable workspace) and say 'onboard me' again — then I make every edit for you, no terminal. Want a no-save preview first? I can build the report and show it without writing anything." Never let the "edits in your repo" promise stand when writes will fail here, and never fall back to asking a non-technical user to run a push or other CLI command (it breaks the no-terminal promise).
- **Check whose repo this is — before the interview, not after the scan.** Run `git remote get-url origin`: if it points at the original template (`fc1206/pulse`) rather than the user's own copy, STOP here. Everything would build locally, but the landing step would fail — they have no push access to the template, and their competitor research has no business near a public repo they don't control. Say it plainly and fix it first, all GUI: have them create their own **private** copy ("Use this template" on the Pulse page, or a fresh private repo at `https://github.com/new`), then re-point origin for them (`git remote set-url origin <their-url>`) and continue — nothing they've said so far is lost. **Never push branches or open PRs/issues against the upstream template** — that would publish their private data.
- **This writes into their repo.** Onboarding generates config files and their **first landscape scan** (a real registry of competitors) *into this repository*. Confirm they want a real setup now, not just a look around. If they're only exploring, stop here and offer to walk them through it instead.
- **The repo is theirs — branch first.** Recommend they let you work on a branch (e.g. `onboarding`) so the changes are easy to review or undo. The repo is theirs to keep, edit, or discard.
- **Set expectations.** The first scan is a **deep map — the full query battery in one pass**: typically **60–150 companies in a dense software category, 20–60 in a leaner niche** (services, agencies, regional lanes), across direct rivals, incumbents bundling this, and adjacent players. Never promise "hundreds." What day one does *not* deliver is deep metadata: founding year, HQ, and stage come back `unknown` unless the evidence page states them; that's honest, not broken, and the next weeks of scans enrich the thin profiles. And a scan you run for them here is a **one-off local scan — not the scheduled GitHub Actions autopilot** (that's the optional last step).

## 0.5 Name the target — STOP until confirmed
Before any web research, any question round, or `/setup`, establish **which company or product this radar is for**. This is the single most expensive thing to get wrong.
- **Ask, in one line:** "What's the company or product this radar is for? Give me its name and website."
- **Never infer the target** from the repo name, open files, loaded skills, or anything earlier in this session. The repo a fresh forker opens is the generic template — it tells you nothing about their company, and a guess from session context is exactly how onboarding ends up researching the wrong company. Don't volunteer a guess; ask.
- **Do not run any search, fetch, or `/setup` step until they answer.** This gate is a hard stop. Everything downstream — the competitor discovery inside `/setup` — keys off the company they name here, not anything you assumed.

## 1. Interview + configure → run `/setup`, then map the landscape
Execute the `/setup` command in full: interview them about their company, lane, and known competitors; capture their **brand** (company name, accent color); write `config/context.md`, `config/rubric.md`, `config/queries.md`, `config/brand.json`; and run the deep-map first scan (setup step 3).
**`config/context.md` is the highest-leverage file** — a vague lane produces a useless brief. Push for sharpness: the exact promise a rival must threaten, the 3–5 axes that decide who competes, and the *wrong-vocabulary* competitor they're most likely to miss.

**The first scan is the landscape map — the full battery, not a trickle.** After the interview, plan it with `python3 scripts/plan_run.py --seed` (every query block at once, marked seed mode). Tell them what's coming, in one line: *"I'll map your whole landscape now — direct rivals, incumbents bundling this, and adjacent players. In a dense category that's 60–150 companies; in a leaner niche more like 20–60. Takes about 20–30 minutes; go get a coffee and I'll have the map when you're back."* Then run it:
- **If this surface supports subagents / parallel tasks:** fan out **one researcher per query block, in parallel**, each returning candidates; merge and dedupe by domain before validation.
- **If it doesn't:** run the blocks sequentially and reset the time expectation honestly (closer to an hour) — never quietly skip blocks to hit the estimate.
- **Sweep every cluster in `config/clusters.json`, not just direct lookalikes.** The incumbents whose suites bundle this, the platforms/infra rivals build on, the adjacent players the buyer confuses them with — all of it. On the radar this template came from, direct rivals were barely a third of the day-one map; a rivals-only map is an incomplete one. Harvest the category's *real* directories while you're at it (Block I — Clutch/Behance for agencies and studios, G2/Capterra for SaaS): directories hold the established players your keyword vocabulary misses. If a cluster ends the sweep empty, name it and say why — never silently skip it.

**Seed acceptance standard (state it, then hold it):**
- **Live evidence URL mandatory** — the unchanged hard gate. No URL, no entry; plausible-but-unverifiable → `watch-unconfirmed` in run notes.
- **Same rubric, same tiers** — wider aperture, same bar, never a lower one; a seed row must pass the same scoring any scheduled scan row would.
- **`founded`/`stage`/`hq` = `unknown` unless stated on the evidence page — never inferred.** Fast harvests are where wrong founding years come from; `unknown` is honest, a guessed year is a bug.
- The report marks thin profiles honestly; the next weeks of scans enrich them (the `/scan` enrichment quota exists for exactly this).

**This map IS their first scan.** It's a real, dated scan run (`runs/<today>/`) — verified, merged, and digested like any other scan. Don't run a second `/scan` today — that would create a duplicate same-date run, double-count the seed, and make the activity chart look like a fake ramp. One run today, labeled as the seed/first scan. The next real scan happens on the next scheduled day (Mon/Thu under autopilot), or whenever they ask you to run `/scan` again — not minutes from now.

This first scan uses **your own model access** (you, the agent, are running it), so they get a real result **without setting up any key or secret**.

## 2. Show them the payoff
Render the deliverable:
```bash
python3 scripts/render_report.py
```
`data/report.html` is their dashboard. **Surface it however the environment allows:** an IDE or **Cowork preview panel** shows the HTML automatically; on a local Mac you can `open data/report.html`. If you're in a headless / cloud session with no preview, that's fine — don't fail on `open`. Either way, walk them through the **digest** in one short paragraph: the one or two things that moved (or an honest "no actionable signal") from this first map and what to do about each, and tell them the dashboard lives at `data/report.html` (and updates every scan). Be honest about depth — the map is wide, but some profiles are thin (`unknown` founded/HQ/stage) until later scans enrich them. This is the moment they understand the product — make it land.

**Name what they're looking at — proactively, before they ask.** In one line: "This is your **landscape map** — the full discovery battery in one pass, N companies. Some profiles are thin on purpose: founding year, HQ, and stage are only recorded when a live source states them, and the twice-a-week scans fill those in while catching funding rounds, acquisitions, and new entrants." Don't make them ask "is this everything?"

## 3. Autopilot — the one manual step, and it's a GUI
For the radar to run itself twice a week it needs their Anthropic API key stored as a GitHub **secret**. This is the only thing they do by hand (plus, at most, a two-click PR merge in item 4 if you can't push), and it is all **graphical web pages — never a terminal or text editor**:
1. Get their repo from `git remote get-url origin` and derive `<owner>/<repo>` — handle both `https://github.com/owner/repo(.git)` and `git@github.com:owner/repo.git`. **If origin is not a GitHub URL** (a local path, or they haven't pushed the repo to GitHub yet), autopilot can't run — GitHub Actions needs the repo on GitHub first. Offer to help them create it at `https://github.com/new` and push, or skip autopilot for now (they keep running `/scan` through you, key-free). Never fabricate the link from a non-GitHub origin.
2. Give them this exact deep link (GitHub's secrets GUI, name pre-filled):
   `https://github.com/<owner>/<repo>/settings/secrets/actions/new?name=ANTHROPIC_API_KEY`
   Tell them: open it → paste their key from **console.anthropic.com** (Settings → API Keys) → click **Add secret**. ~30 seconds.
3. Once they confirm it's added, **you** enable the schedule for them: uncomment the `schedule:` block at the top of `.github/workflows/scan.yml` and commit (on the `onboarding` branch, like everything else).
4. **Land the onboarding branch on `main` — autopilot does nothing until this happens.** The Mon/Thu cron only fires from the repo's **default branch**: until the workflow's `schedule:` block, the config, and the seed scan are on `main`, the schedule is a no-op. Say that plainly. And landing the seed is sanctioned, not a rule break: it went through `validate_merge.py`/`validate_digest.py` with `RADAR_ALLOW_WRITE=1`, so it IS the validated lineage — the one exception to "local runs never reach `main`" (CLAUDE.md rule 7 is satisfied, not bypassed).
   - **You can push:** push the `onboarding` branch and merge it to `main` for them (merge, never `git reset`), then confirm `main` has the seed + workflow.
   - **You can't push from this session:** the compare deep link (`https://github.com/<owner>/<repo>/compare/main...onboarding` → **Create pull request** → **Merge pull request**) only works if the `onboarding` branch actually reached GitHub — offer it ONLY after confirming the branch exists on the remote (`git ls-remote --heads origin onboarding`). If the push itself failed, a compare link 404s and strands them; instead say honestly that this session lacks GitHub write access, and tell them the no-terminal recovery: reopen the repo in a surface with their GitHub account connected (or connect it in this assistant's settings) and say *"land my onboarding branch"* — everything is saved locally and nothing is lost.
- If they're not ready for autopilot, that's fine — they can run `/scan` manually anytime, key-free, through you, and merge the onboarding branch whenever they choose (or ask you to).

## 4. Done — recap
Summarize: their radar is set up (N companies on the landscape map, here's this scan's brief in one line), what happens next — if they added the key AND the onboarding branch is merged to `main`, autopilot scans Mon/Thu on its own; if either is missing (no key, or the work is still sitting on the `onboarding` branch), it stays a manual `/scan` through you — the schedule only fires from the default branch, so nothing runs on its own until the merge lands — and the one habit that keeps it sharp: keep `config/context.md` current as their strategy moves. Privacy note: keep the repo **private** if their competitor set is sensitive; the report is a local file, not published anywhere unless they choose to.
